# Описание сервисов (Dating Bot)

Ниже — логические сервисы системы и их ответственность. Границы можно реализовать как отдельные процессы/репозитории или как модули в одном backend — на этапе проектирования важна декомпозиция по зонам ответственности.

## 1. Telegram Gateway (Bot Service)

- Приём обновлений от Telegram Bot API (long polling или webhook).
- Маршрутизация команд (`/start`, навигация по анкетам, лайк/пропуск, настройки).
- Идентификация пользователя по `telegram_user_id`; передача событий во внутренние API или в очередь сообщений.
- Отправка сообщений, клавиатур, медиа пользователю.

## 2. Identity & Session Service

- Связка аккаунта приложения с Telegram ID.
- Опционально: сессии, ограничение частоты запросов с одного пользователя.

## 3. Profile Service (Анкеты)

- CRUD профиля: пол, возраст, город, интересы, текст анкеты, фото.
- Загрузка изображений: выдача presigned URL или проксирование в объектное хранилище (S3-совместимое, например MinIO).
- Расчёт «полноты» анкеты и числа фото для первичного рейтинга.

## 4. Matching & Feed Service

- Подбор кандидатов для показа с учётом фильтров пользователя (возраст, пол, город).
- Запрос предварительно отранжированной выдачи из кэша (Redis) и догрузка следующей «пачки» (например, 10 анкет) при приближении к концу списка.
- Исключение уже просмотренных/заблокированных профилей.

## 5. Rating Service

- **Уровень 1 — первичный рейтинг:** признаки из анкеты, полнота, число фото, совпадение с предпочтениями просматривающего.
- **Уровень 2 — поведенческий:** лайки, пропуски, мэтчи, инициация диалогов, опционально временные паттерны активности.
- **Уровень 3 — комбинированный:** взвешенная модель + доп. факторы (например, рефералы).
- Персистентность агрегатов в отдельной таблице/наборе таблиц; периодический пересчёт (Celery).

## 6. Interaction & Events Service

- Фиксация событий: просмотр, лайк, скип, мэтч, сообщение после мэтча.
- Публикация событий в брокер (Kafka / RabbitMQ / др.) для асинхронной обработки, пересчёта метрик и рейтингов.

## 7. Cache Layer (Redis)

- Кэш предварительно отранжированных списков профилей на пользователя (ключи по `user_id` + версия/смещение).
- Краткоживущие данные сессии, rate limiting, блокировки при пересчётах (по необходимости).

## 8. Media Storage

- Хранение фото анкет в S3-совместимом API (в т.ч. MinIO для локальной разработки).
- В БД — только метаданные и ключи объектов.

## 9. Background Workers (Celery)

- Пересчёт рейтингов по расписанию или по событиям из очереди.
- Очистка устаревшего кэша, агрегации для аналитики.

## 10. Observability

- Структурированное логирование запросов и бизнес-событий.
- Метрики (latency, ошибки, размер очередей, hit rate кэша) — интеграция с Prometheus или аналогом на этапах реализации.
